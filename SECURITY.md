# Security Policy

Revo-Auth is an authentication platform, so we treat security reports as first-class
issues. Thanks for taking the time to disclose responsibly.

## Reporting a vulnerability

- Email: **security@revo-auth.dev** (PGP encrypted if sensitive - key below).
- Please do **not** open a public GitHub issue, draft PR, or discussion thread for
  anything that could be exploited before a patch ships.
- Include: affected version(s), reproduction steps, impact assessment, and any
  proof-of-concept code. Screenshots and HTTP captures help.

We use GitHub's private vulnerability reporting as a fallback - open a
[security advisory](https://github.com/revo-auth/revo-auth/security/advisories/new)
if email is inconvenient.

## Response SLA

| Stage | Target |
| --- | --- |
| Acknowledgement of report | within **72 hours** |
| Triage + severity assignment (CVSS v3.1) | within **7 days** |
| Patch released for supported versions | within **30 days** for High/Critical |
| Public disclosure + CVE request | coordinated with reporter after patch |

We practice coordinated disclosure; if you need an embargo extension for a downstream
dependency, ask.

## Supported versions

| Version | Status |
| --- | --- |
| `0.x` (current) | Supported - security fixes on the latest minor. |
| Pre-1.0 older minors | Best effort, no guarantees. |

Once `1.0` ships, the latest two minor lines receive security patches for 12 months.

## Scope

**In scope:**

- `apps/server` - Rust auth server (Axum, Postgres, Redis).
- `packages/sdk-core`, `packages/sdk-sveltekit`, `packages/ui-sveltekit`.
- `packages/cli` - scaffolding CLI and its generated templates.
- The distroless Docker image published to GHCR.

**Out of scope:**

- User-scaffolded application code generated by the CLI after customization.
- Third-party OAuth/OIDC providers; report upstream.
- Vulnerabilities in dependencies already tracked by `cargo audit` / `pnpm audit`
  with an unreleased patch (please still let us know).
- Social engineering, physical attacks, and denial-of-service against personal
  infrastructure.

## Safe harbor

We will not pursue legal action against researchers who:

1. Make a good-faith effort to avoid privacy violations, service degradation, or
   data destruction.
2. Only interact with accounts and data they own or have explicit permission to test.
3. Report the issue privately and give us a reasonable window to remediate.
